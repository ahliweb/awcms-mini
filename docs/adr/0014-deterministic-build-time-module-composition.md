# ADR-0014 — Komposisi modul deterministik saat build-time untuk registry base dan aplikasi turunan

- **Status:** Accepted
- **Tanggal:** 2026-07-13
- **Pengambil keputusan:** @ahliweb
- **Terkait:** Issue #740 (epic #738 `platform-evolution`, Wave 1), Issue #739/ADR-0013, ADR-0011, ADR-0012, `docs/awcms-mini/21_module_admission_governance.md`, `docs/awcms-mini/derived-application-guide.md`, `src/modules/module-management/domain/module-dependency-graph.ts`, `src/modules/module-management/domain/module-composition.ts`

## Konteks

Registry modul (`src/modules/index.ts`) adalah satu array statis yang di-review lewat PR normal (ADR-0012 §7). Sampai saat ini, satu-satunya cara aplikasi turunan (AWPOS, Smart School Portal, dst. — `derived-application-guide.md`) menambahkan modul domainnya sendiri adalah dengan mem-fork/vendor repo base ini lalu **mengedit langsung** `src/modules/index.ts` miliknya sendiri (ADR-0013 §5, "pola yang sudah berjalan"). Ini aman selama hanya ada sedikit repo turunan, tapi tidak scale: setiap repo turunan harus menyelesaikan merge conflict di file yang sama setiap kali menyerap update dari base, tidak ada satu titik integrasi yang jelas untuk direview, dan tidak ada mekanisme yang mencegah nomor migration/nama capability/module key aplikasi turunan bertabrakan dengan base.

ADR-0013 §5 secara eksplisit mencatat celah ini dan sengaja **tidak** mendesainnya: _"Kandidat mekanisme deterministik pengganti ('deterministic build-time module assembly', Wave 1 epic #738) belum ada dan tidak didesain oleh ADR ini — ADR ini hanya menyatakan bahwa begitu mekanisme itu ada, ia harus tetap tunduk pada arah DAG §1 dan tidak pernah mengizinkan aplikasi turunan mengedit registry base secara langsung."_ Issue #740 adalah mekanisme itu. Sebelum mendesain, dibaca sebagai ground truth: `docs/adr/0013-extension-layers-and-boundary-model.md` (lapisan ekstensi, arah dependency DAG lintas-repo, no-shared-table-write), `docs/awcms-mini/21_module_admission_governance.md` (lima kategori admission, §7 larangan runtime loading), `src/modules/_shared/module-contract.ts` (kontrak `ModuleDescriptor` yang tidak diubah bentuknya, hanya diperluas secara aditif), dan `src/modules/module-management/domain/module-dependency-graph.ts` (validator DAG registry-wide yang sudah ada, Issue #680, dipakai ulang bukan diduplikasi).

## Keputusan

### 1. API komposisi tiga bagian

- **`listBaseModules()`** (`src/modules/index.ts`) — mengembalikan array 16 modul base, tidak berubah bentuk/urutan/isi dari sebelum issue ini.
- **Titik ekstensi build-time tunggal, `src/modules/application-registry.ts`** — satu-satunya file yang boleh diedit sebuah repo turunan. Repo base ini sendiri mengirim `applicationModuleRegistry: ApplicationModuleRegistry | undefined = undefined`. Sebuah repo turunan mengganti nilai itu dengan `ApplicationModuleRegistry` miliknya sendiri (`{ id, modules, migrationNamespace? }`, tipe baru di `_shared/module-contract.ts`) — masih 100% TypeScript compile-time biasa, di-resolve `bun run build`/`bun run typecheck` seperti import lain, **bukan** runtime discovery/file upload/package scanning/`eval` (menegaskan ulang guardrail ADR-0013 §9/doc 21 §7, tidak melonggarkannya).
- **`composeModuleRegistry()`** (`src/modules/module-management/domain/module-composition.ts`) — menggabungkan base + application registry lalu memvalidasinya, mengembalikan `{ valid: true, registry }` atau `{ valid: false, registry, issues }`.

### 2. Dua fase yang sengaja dipisah: merge (selalu sukses) vs validate (bisa gagal)

`mergeModuleRegistries(base, application)` adalah concatenation murni (base dulu, lalu modul application registry, urutan masing-masing dipertahankan) — **tidak pernah melempar**. `src/modules/index.ts` HANYA memanggil fungsi ini, persis meniru arsitektur yang sudah ada (`index.ts` selalu data murni; `listModules()` sebelum issue ini adalah `return modules` tanpa validasi apa pun — validasi selalu langkah eksplisit terpisah yang dipanggil script/test, preseden `validateModuleDependencyGraph`, Issue #680). Konsekuensi langsung: karena `applicationModuleRegistry` di repo base ini SELALU `undefined`, `mergeModuleRegistries(base, undefined)` adalah pass-through murni (`[...base]`) — **registry efektif default build base identik byte-demi-byte dengan sebelum issue ini**, dibuktikan test (`tests/unit/module-composition.test.ts`'s "listModules() is byte-identical to listBaseModules()").

`validateComposedModuleRegistry()`/`bun run modules:compose:check` (`scripts/validate-module-composition.ts`) adalah mesin aturan sesungguhnya — dipanggil eksplisit oleh CI/test, tidak pernah oleh `index.ts` saat modul di-load.

### 3. Taksonomi kegagalan komposisi (`ModuleCompositionIssue`)

Empat dipakai ulang apa adanya dari validator DAG registry-wide yang sudah ada (Issue #680), dijalankan atas hasil merge: `self_dependency`, `duplicate_dependency`, `missing_dependency`, `cycle`.

Tujuh baru di issue ini:

Tidak ada issue type khusus untuk katalog permission (namespace-nya sudah `${moduleKey}.${activityCode}.${action}`, tabrakan lintas modul mustahil secara struktural) atau health descriptor (`ModuleHealthContract` cuma dua boolean opsional, tidak ada bentuk untuk divalidasi) — keduanya tetap mengalir ke inventory komposisi (§6) supaya tetap tercermin.

### 4. Konvensi namespace migration

`BASE_MODULE_MIGRATION_NAMESPACE` (`module-composition.ts`) mereservasi `1-899` untuk migration base repo ini (`sql/NNN_awcms_mini_*.sql`) — jauh di atas nomor tertinggi nyata hari ini. `ApplicationModuleRegistry.migrationNamespace` (opsional) mendeklarasikan range yang dimiliki `sql/` milik repo turunan sendiri; komposisi menolak bila beririsan dengan range base. Ini murni perbandingan data yang dideklarasikan (fungsi domain tetap tanpa akses filesystem, konsisten dengan `module-dependency-graph.ts`) — **tidak** membaca file `sql/*.sql` nyata; itu tetap tanggung jawab `scripts/repo-inventory-generate.ts`. Rekomendasi bagi repo turunan: mulai penomoran migration sendiri dari `900`.

### 5. Metadata kompatibilitas deployment profile

`ModuleCompatibilityContract` (`_shared/module-contract.ts`) diperluas secara ADITIF dengan `deploymentProfiles?: readonly ModuleDeploymentProfile[]` (empat nilai yang sama seperti `src/lib/config/registry.ts`'s `DeploymentProfile`, diredeklarasikan lokal supaya file kontrak tetap tanpa dependency). Tidak diisi = kompatibel dengan semua profil (konvensi absence yang sama seperti `minAppVersion`). Komposisi melaporkan `deployment_profile_incompatible` bila sebuah modul mengklaim profil yang tidak didukung salah satu dependency lifecycle-nya. Tidak ada modul base yang mengisi field ini hari ini — perluasan ini nol-risiko terhadap build base yang ada.

### 6. Inventory komposisi deterministik untuk bukti CI/rilis

`buildComposedModuleInventory()` + `bun run modules:composition:inventory:generate`/`:check` (`docs/awcms-mini/module-composition-inventory.json`) — snapshot JSON deterministik (diurutkan per `key`, tanpa timestamp wall-clock, pola freshness yang sama seperti `repo-inventory.md`) berisi setiap modul (key/versi/status/type/source base-atau-application/dependencies/capability/hitungan permission-navigation-job/health/deploymentProfiles) plus namespace migration base dan application. Wired ke `bun run check`.

### 7. Konsumen registry lama otomatis jadi compose-aware

`listModules()` (dipakai `bun run modules:sync`, `bun run modules:dag:check`, `bun run repo:inventory:generate`, dan semua service `module-management`) sekarang mengembalikan hasil `mergeModuleRegistries(baseModules, applicationModuleRegistry)`, bukan array base langsung. Karena SEMUA consumer registry yang sudah ada memanggil `listModules()` (dikonfirmasi lewat audit source sebelum implementasi — tidak ditemukan sumber array modul kedua/duplikat), tidak satu pun butuh perubahan kode untuk otomatis menjadi compose-aware begitu sebuah repo turunan mengisi `application-registry.ts` miliknya — ini yang dimaksud acceptance criteria "module-management sync consumes the composed registry, not an unrelated duplicate source".

## Konsekuensi

- **Positif:** Repo turunan tidak lagi perlu mengedit `src/modules/index.ts`/`module.ts` base mana pun — satu titik integrasi (`application-registry.ts`) yang jauh lebih kecil untuk di-review dan di-merge ulang tiap kali menyerap update base, mengurangi kelas konflik yang disebut Context di atas.
- **Positif:** Konvensi namespace migration mencegah SATU kelas tabrakan migration (penomoran) lewat deteksi di compose-time, sebelum migration file apa pun ditulis — bukan solusi penuh (tidak membaca `sql/` nyata), tapi guardrail nyata yang sebelumnya tidak ada sama sekali.
- **Positif:** Build base default (registry efektif) tidak berubah sama sekali — dibuktikan test langsung terhadap `listModules()`/`listBaseModules()` yang nyata, bukan cuma diklaim.
- **Negatif/trade-off:** Dua tipe kontrak baru (`ApplicationModuleRegistry`, `ModuleMigrationNamespace`) dan satu field `ModuleCompatibilityContract` yang diperluas — aditif murni, tidak breaking, tapi menambah permukaan kontrak `_shared/module-contract.ts` yang harus dipahami penulis modul baru.
- **Negatif/trade-off:** Penegakan "no shared-table write" lintas-repo (ADR-0013 §6) **tetap** tidak diotomasi oleh mekanisme ini — komposisi memvalidasi identitas/dependency/capability/migration-namespace/deployment-profile di level DESKRIPTOR, bukan akses tabel nyata lintas proses/repo terpisah. Tetap sebagaimana dicatat ADR-0013: kepatuhan lintas-repo bersandar pada review manual + checklist admission, bukan diklaim ditegakkan mesin.
- **Netral:** Tidak mengubah union `ModuleType`, tidak mereklasifikasi modul mana pun, tidak menambah migration SQL baru (komposisi murni level deskriptor TypeScript, in-memory), tidak menambah endpoint/event baru.

## Alternatif yang dipertimbangkan

- **Runtime plugin loading / dynamic import dari path yang bisa dikonfigurasi env var** — ditolak eksplisit: bertentangan langsung dengan larangan keras doc 21 §7 dan ADR-0012 §7 (tidak ada runtime code loading tanpa ADR baru yang men-supersede ADR-0001/ADR-0002; issue ini tidak mengajukan supersede itu).
- **Satu array gabungan tanpa validasi khusus (merge saja, tanpa `validateComposedModuleRegistry`)** — ditolak: tidak memenuhi acceptance criteria "duplicate key/missing dependency/cycle/missing capability/conflicting provider/prohibited Core override each fail with actionable diagnostics".
- **Filter `prohibited_base_override` hanya untuk modul `type: "core"`/`"system"` (menyalin kalimat literal guardrail apa adanya)** — ditolak: `type` tidak konsisten diisi di registry base hari ini (doc 21 §8 R1), aturan longgar akan bisa dilewati modul base yang kebetulan tidak mengisi `type`. Aturan lebih ketat (semua modul base) dipilih karena strict superset dari guardrail yang diminta dan tidak melarang skenario turunan yang sah.
- **Validasi migration namespace membaca file `sql/*.sql` sungguhan di dalam fungsi domain komposisi** — ditolak: melanggar konvensi "domain = murni, tanpa I/O" yang sudah ditetapkan `module-dependency-graph.ts`; perbandingan data yang dideklarasikan sudah cukup untuk mendeteksi kelas tabrakan yang paling umum (dua repo turunan tidak berkoordinasi), pembacaan filesystem nyata tetap domain `repo-inventory-generate.ts`.
- **Menaruh mesin validasi di `_shared/` alih-alih `module-management/domain/`** — ditolak: akan membalik arah dependency (`_shared/module-contract.ts` sengaja tanpa import apa pun sejak awal; `module-management/domain/module-dependency-graph.ts` sudah menjadi pemilik yang mapan untuk validasi registry-wide, dipakai ulang bukan dipindah).
